## Project Context & Objectives

Build a production-grade, event-driven Inventory Tracking System named **"Divine Hindu Inventory Tracking System"**. The core objective is to provide **real-time, SKU-level visibility** across multiple warehouses and channels with an **immutable audit trail (Ledger)**.

The system must strictly prevent overselling and duplicate counts using pessimistic row-level locking and idempotency keys.

### Tech Stack

* **Backend:** FastAPI (Python 3.11+) with SQLAlchemy 2.0 (async preferred).
* **Database:** PostgreSQL 15+.
* **Task Queue & Cache:** Redis + Celery + Celery Beat.
* **Frontend:** React (Vite, TypeScript, Tailwind CSS, Material UI or shadcn/ui).
* **Reporting:** Pandas / NumPy.
* **Notifications:** Dedicated Email service (`smtplib` / AWS SES).

---

## 1. Backend Specifications (FastAPI & PostgreSQL)

### Data Models & Constraints

Implement the following schema layout using SQLAlchemy. Ensure soft deletes (`is_active`) on master data and enforce appropriate cascading rules.

* `products`: Master SKU details (`id`, `sku_code` (Unique Index), `product_name`, `category`, `brand`, `unit`, `minimum_stock`, `reorder_stock`, `is_active`).
* `warehouses`: Multiple locations (`id`, `warehouse_name`, `location`, `warehouse_type` e.g., Main, Quarantine, Damaged).
* `inventory_balance`: Fast lookup caching current state (`product_id`, `warehouse_id`, `available_qty`, `reserved_qty`, `damaged_qty`, `expired_qty`, `returned_qty`). Include a **Composite Unique Index** on `(product_id, warehouse_id)`.
* `stock_movements`: Delta tracking (`id`, `movement_id` (UUID, Unique), `product_id`, `warehouse_id`, `movement_type` [INWARD, SALE, RETURN, DISPATCH, TRANSFER_IN, TRANSFER_OUT, DAMAGE, EXPIRED, ADJUSTMENT], `qty`, `reference_type` [order, return, reconciliation], `reference_id`, `created_at`).
* `inventory_ledger`: **Immutable ledger table** (`id`, `product_id`, `warehouse_id`, `movement_id`, `opening_qty`, `change_qty`, `closing_qty`, `transaction_time`).
* `sales_orders` & `sales_order_items`: Basic order tables to track channel and quantity updates.
* `returns`: Tracks customer returns, including `qc_status` (PENDING, GOOD, DAMAGED, EXPIRED).
* `stock_reconciliation`: Captures manual cycle counts (`system_qty`, `physical_qty`, `variance`, `status` [PENDING, APPROVED]).

### Core Backend Logic Requirements

1. **Concurrency Control:** When a sales event occurs, use `SELECT ... FOR UPDATE` via SQLAlchemy on the `inventory_balance` row to serialize updates and prevent race conditions.
2. **Idempotency Engine:** Implement a middleware or service check that verifies if an incoming `event_id` or `movement_id` has already been processed in `stock_movements`. If it exists, gracefully skip processing.
3. **The Immutable Rule:** Never update `inventory_balance` directly without creating a corresponding `stock_movements` record and appending a line to the `inventory_ledger`.

## 2. Background Workers & Reporting (Celery + Redis)

Configure a Celery instance with Redis as the broker to handle long-running transactions and cron schedules.

### Tasks to Implement

* **Low Stock Alert Engine (Run every 15 mins):** Queries `inventory_balance` where `available_qty <= minimum_stock`. For any matching SKU, trigger an automated Email notification detailing current levels vs thresholds.
* **Daily Closing Stock Report (Run daily at 11:59 PM):** Computes snapshot balances utilizing Pandas:

$$\text{Closing Stock} = \text{Opening Stock} + \text{Inward Stock} - \text{Outward Stock}$$



Saves state and emails a consolidated spreadsheet/PDF to management.
* **Automatic Variance Resolution:** A background task triggered when a Manager clicks "Approve Variance" on the frontend. It calculates the delta and inserts an `ADJUSTMENT` movement type to reconcile system metrics with physical counts.

## 3. Frontend Specifications (React + TypeScript)

Create a clean, dashboard-driven Single Page Application (SPA) structured as follows:

### Required Views & UI Modules

1. **Dashboard Hub:**
* Display key metrics: Total Inventory Value, Low Stock SKU Counts, Today's Returns, and Pending Dispatches.
* Include interactive tabular alerts for items requiring replenishment.


2. **Inventory Ledger & Stock Level Grid:**
* A searchable, paginated data table showing all active SKUs across warehouses.
* An expandable detailed view showing the audit trail (`inventory_ledger`) for a specific SKU.


3. **Return Operations Desk (QC Workflow):**
* An interface to ingest new customer returns, pushing items to the "Quarantine Zone" instantly.
* A simple, three-button evaluation panel for warehouse personnel: **[Pass QC -> Move to Available]**, **[Fail QC (Damaged) -> Move to Damaged]**, or **[Fail QC (Expired) -> Move to Expired]**.


4. **Stock Reconciliation Suite:**
* Form module allowing team members to type in a `Physical Count` after scanning a warehouse partition.
* Variance Calculator showing system mismatch ($Variance = Physical - System$).
* An **"Approve Adjustment"** workflow button restricted to administrative roles.
