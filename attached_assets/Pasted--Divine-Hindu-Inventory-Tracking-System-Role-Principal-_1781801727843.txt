# Divine Hindu Inventory Tracking System

**Role:** Principal Software Architect
**Goal:** Real-time SKU-level inventory visibility with accurate stock reconciliation across sales, returns, warehouse operations, damaged stock, and dispatch.

---

# 1. High-Level Architecture

## Architecture Principles

1. **Inventory Ledger is the Single Source of Truth**

   * Never update stock directly.
   * Every stock change creates a ledger transaction.

2. **Event-Driven Inventory**

   * Sales, returns, dispatches, inward receipts, and damages generate events.
   * Events create stock movement records.
   * Stock balances are derived from movements.

3. **ACID Transactions**

   * PostgreSQL ensures inventory consistency.
   * Row-level locking prevents overselling.

4. **Real-Time Visibility**

   * APIs update stock instantly.
   * Dashboard reads current balances.

# Daily Closing Stock Algorithm

Python Pseudo Code

```python
def calculate_closing_stock(product_id, date):

    opening = get_previous_day_closing(product_id)

    inward = sum_movements(
        product_id,
        ["INWARD", "RETURN", "TRANSFER_IN"]
    )

    outward = sum_movements(
        product_id,
        [
            "SALE",
            "DISPATCH",
            "TRANSFER_OUT",
            "DAMAGE",
            "EXPIRED"
        ]
    )

    closing = opening + inward - outward

    return closing
```

# 5. Channel Alignment Strategy

Challenge:

```text
Amazon Order

Website Order

POS Sale

All arrive simultaneously.
```

Potential issue:

Overselling.

---

## Solution

### Row-Level Locking

```sql
SELECT *
FROM inventory_balance
WHERE product_id = ?
FOR UPDATE;
```

This locks inventory until transaction completes.

---

## Transaction Flow

```python
BEGIN

Lock Inventory Row

Check Available Qty

Reserve Qty

Create Movement

Create Ledger

Update Balance

COMMIT
```

Guarantees:

* No duplicate deductions
* No negative inventory
* No race conditions

```python
if event_exists(event_id):
```

# Final Production Architecture

**Backend:** FastAPI
**Database:** PostgreSQL
**ORM:** SQLAlchemy
**Cache/Queue:** Redis
**Workers:** Celery + Celery Beat
**Reporting:** Pandas + NumPy
**Notifications:** Email + WhatsApp + Slack
**Dashboard:** React (or Streamlit MVP)
**Mobile Warehouse App:** React Native
**Infrastructure:** Docker + Nginx + Ubuntu VPS/AWS EC2

This architecture can comfortably support **50,000+ SKUs, multiple warehouses, real-time stock updates, audit-grade traceability, automated reconciliation, and D2C/eCommerce integrations** while keeping the workflow simple enough for warehouse operators.
